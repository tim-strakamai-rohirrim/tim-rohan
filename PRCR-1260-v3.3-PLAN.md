## Problem Statement

The **Document Viewer Panel** in the Compliance List Creation workflow (`document-viewer-panel.component`) currently displays **mock/hardcoded text lines** instead of actual document content. The `mapProjectDocumentsToViewerDocuments()` utility in `compliance-document.utils.ts` maps project documents onto `MOCK_COMPLIANCE_DOCUMENTS` line data (with a TODO to replace it).

Users need to:

1. View the **real content** of each uploaded source document in the right-hand panel, with layout that **matches the look of the original document**.
2. **Highlight text** within the rendered document and create compliance items (tagging) from the selection — even when auto-tagging has not been run.
3. **Edit both auto-tagged and manually created compliance items** (title, text, etc.) while viewing the highlighted source passage for context.
4. Switch between multiple documents via the existing dropdown.
5. **Add new documents** to an existing project from the overview page's Document Library (post-wizard), with auto-tagging running automatically on only the new documents after conversion completes.
6. **Delete documents** from the library, with all associated compliance items automatically removed from the database.

The viewer must support **PDF, DOCX, and XLSX** files. Docling already converts all three formats to HTML uniformly, so one rendering path handles all file types.

### Document lifecycle context

- The **wizard** (Create → Preview → Finish) already triggers auto-tagging on all uploaded documents when the user clicks Finish (`onFinish()` calls `processDocuments(projectId, true)` fire-and-forget). This behaviour is correct and does not change.
- After finishing the wizard, the **overview page** includes a Document Library section. Documents uploaded here must be auto-tagged automatically after upload — there is no "Finish" button.
- Deleting a document from the overview page must cascade-delete all associated compliance items (already handled by `ON DELETE CASCADE` in the DB).

## Key Architectural Observations

### How HTML is produced today

1. **Upload**: Frontend uploads file via `POST /compliance/projects/:project_id/documents` → stored in MinIO at `compliance_documents.minio_object_key`. **Post-merge:** the upload endpoint (`uploadSourceDocument`) now also resets `autotag_processing` to `OutlineStatus.n` (`'Not Started'`) and emits `compliance.auto.tag`, so new documents are auto-tagged immediately after upload.
2. **Auto-tag trigger**: `POST /compliance/projects/:project_id/documents/process` → NestJS emits `compliance.auto.tag` event → `ComplianceListener.autoTagHandler` sends one Service Bus `AutoTagRequestMessage` per document. **Post-merge:** the listener now skips documents where `processingStatus !== PENDING_EXTRACTION || taggable_doc_id != null`, preventing double-submission. It also sets `processingStatus = EXTRACTING` before the Service Bus call.
3. **Python worker**: Downloads file from MinIO → Docling converts to HTML → `strip_html_metadata` removes `<style>`, `<script>`, `<head>` → segments HTML → classifies tags → **uploads stripped HTML** to MinIO at `{parent_path}/{output_folder}/{stem}.html` (where `output_folder` defaults to `"output"` via `_resolve_output_folder`, configurable per-message or via `MINIO_TAGGING_FOLDER_OUT`) → sends `AutoTagComplete` with `converted_html_url`.
4. **NestJS handler**: `AutoTagCompleteHandler` persists tags → emits `tagging.auto-tag-complete` event with `convertedHtmlUrl`.
5. **Compliance listener**: `handleAutoTagComplete` marks document ready/failed, loads `DocumentTag` rows, creates `ComplianceItem` rows — but **does not store `convertedHtmlUrl`**.

### Tag config system

The `tag_configs` table stores one row per product code (unique constraint on `product_code`). Each row has `tag_schema` (JSONB), `tag_ui` (JSONB, nullable), `segmentation_strategy`, `prompt_name`, and `rule_pattern`. The backend serves tag configs via `GET /tagging/tag-configs?product_code=...`.

**[v3.4 — PRCR-1421 landed]** The `tag_ui` column (added by PRCR-1421) holds a normalized UI-only display contract (`{ tags: [{ id, name, color }] }`) separate from `tag_schema` (which drives backend classification). All four products are seeded with `tag_ui` — compliance has `{ "tags": [{ "id": "compliance_item", "name": "Add Compliance Item", "color": "#CCE5FF" }] }`. The compliance `tag_schema` still has **no** `tags` or `categories` array (only pipeline fields), but this no longer matters because `doc-shell` reads `tag_ui` first.

On the frontend, `TaggingService` maintains a signal-based in-memory cache (`tagConfigsByProduct`) populated by callers via `setTagConfig(productCode, config)` / `clearTagConfig(productCode)`. Consumers read via `getTagConfigSnapshot(productCode)`. `PostLoginEffectsService` currently preloads tag configs for `TEMPLATE_GENERATOR` and `PROPOSAL_WRITER` at login — **not** `COMPLIANCE` (compliance is a separately-gated feature, so its config should be loaded lazily). **[v3.4]** `TaggingService` now resolves display metadata from `tag_ui` first, falling back to `tag_schema` only when `tag_ui` is null.

The `doc-shell` component accepts a `[tagConfig]` input (type `TagConfigResponse | null`). It does **not** inject `TaggingService` — it relies entirely on the input. **[v3.4]** Inside `buildMenuConfig()`, the shell now tries `tag_ui.tags[]` first via `resolveTagUiTags()`. Only if `tag_ui` is absent/empty does it fall back to `resolveSchemaTags(tagSchema)` which checks `tagSchema.tags` then `tagSchema.categories`. Since compliance `tag_ui` is seeded, `doc-shell` will find it and build the context menu correctly — no `tag_schema` modification is needed.

### Offset semantics

`compliance_items.document_start_line` / `document_end_line` are populated from `DocumentTag.start_offset` / `end_offset` via the compliance listener. Despite the "line" naming, these are **character offsets into the stripped HTML string**, not line numbers. The same offset space applies to the HTML served by the new content endpoint.

### Existing cascade constraints (DB)

- `compliance_items.source_document_id → compliance_documents(id) ON DELETE CASCADE`
- `compliance_project_documents(document_id) → compliance_documents(id) ON DELETE CASCADE`
- `compliance_checks.compliance_item_id → compliance_items(id) ON DELETE CASCADE`
- `compliance_item_evidence.response_document_id → compliance_documents(id) ON DELETE CASCADE`

Deleting a `compliance_documents` row automatically removes all associated items, checks, and evidence.

### Existing document deletion via PATCH

The `PATCH /projects/:id` endpoint (`updateProject`) already supports batch document deletion via a `documentsToDelete` property on the DTO. The wizard uses this when editing a project to remove documents the user un-selected. The implementation validates project-scoping, deletes `ComplianceDocument` rows (DB cascade handles items, checks, evidence), and cleans up `TaggableDocument` rows. It does **not** delete the corresponding MinIO objects (original files or converted HTML). This is intentional — files in MinIO are retained for traceability even after the database record is deleted.

The `PATCH /projects/:projectId/responses/:responseId` endpoint also has a `documentsToDelete` property, but that only unlinks response documents from a response (deletes `compliance_response_documents` join-table rows) — it does **not** delete the `ComplianceDocument` row itself.

## Assumptions

1. Docling-generated stripped HTML is **safe to render**. The shared `app-doc-shell` component already handles sanitisation via DOMPurify + `bypassSecurityTrustHtml`, matching the Proposal Writer's approach.
2. The stripped HTML uploaded by the Python auto-tag worker is the **same** HTML we serve in the viewer — no secondary conversion needed.
3. The viewer only needs to display document content **after** auto-tagging has been triggered (wizard Finish or overview upload). A "Processing…" placeholder for in-progress documents is acceptable for v1.
4. The existing `pageNumber` / `totalPages` fields on `ComplianceSourceDocument` can be dropped — Docling HTML does not inherently have page breaks.
5. The shared `app-doc-shell` component's offset algorithm (`buildFlattenedOffsetsByTextNode`) is compatible with the Python auto-tag pipeline's `extract_text_with_mapping()` — both use the same block selectors and separator. This is explicitly documented in the Python code comments.
6. `app-doc-shell` is exported by `SharedComponentsModule`, which `ComplianceModule` already imports. No additional module import is needed.
7. **Post-merge:** `doc-shell` now rebuilds flattened offsets after each highlight DOM mutation, fixing overlapping highlight rendering. This means compliance items with overlapping offsets will render correctly without custom workarounds.
8. **Post-merge:** `InlineTag.isManualPosition` has been removed from the shared type. The plan previously noted setting `x`/`y` to `0`; this is still correct, but `isManualPosition` no longer exists and should not be referenced.

## Follow-up: Document processing status visibility

> **Status:** Deferred from Phase 8 (v3.5). To be addressed in a separate ticket.

### Problem

In Phase 8 (v3.5), the Overview page Document Library was simplified to match the Create step exactly: four columns (Reference File Name, Size, Type, Actions=delete) with no status badge and no per-row retry button. This is the right call for visual consistency, but it introduces a UX gap: **a user has no visual signal when backend extraction fails on Overview**.

Failure modes that produce `processingStatus = Failed` in `compliance_documents`:

- Docling cannot parse a corrupt or unsupported file
- MinIO or Service Bus transient errors during the auto-tag round trip
- Auto-tag worker timeout

Today's recourse without a retry button: the user must delete the document and re-upload it. Today's recourse without a status badge: the user has no idea anything went wrong — the file appears in the list, they navigate to the compliance items panel later, see no items, and don't know why.

The polling primitives already exist in `ComplianceStateService` (`startProcessingPoll()` / `beginPolling()` / `stopProcessingPoll()`), so observing the `Failed` transition is essentially free.

### Suggested approaches (pick one in the follow-up ticket)

#### Option A — Toast on failure transition (recommended)

When `ComplianceStateService.startProcessingPoll()` observes a document transitioning to `Failed`, emit a snackbar / toast:

> "Failed to process **{document.name}**. Delete and re-upload, or contact support."

Optionally include a "Retry" action on the toast that calls `ComplianceApiService.processDocument(projectId, documentId)`.

Pros:

- Zero impact on the table layout (preserves visual parity with Create step)
- Matches modern web UX for async background tasks
- Reuses existing polling infrastructure
- Retry can be a single CTA on the toast

Cons:

- Toast can be missed if the user is in another tab when it fires
- Stale failures (e.g. discovered on next page load) need a different surface

#### Option B — Subtle inline indicator on failed rows

Add a small red exclamation `mat-icon` prefix to the file name in `app-document-upload`'s internal table for documents whose `UploadedFile` carries an extra `extractionFailed: true` flag. Tooltip on hover explains the failure. Right-click or trailing menu offers "Retry" / "Delete".

Pros:

- Persistent — visible whenever the user looks at the list, not just when the toast fires
- Minimal column-layout impact (single icon, no new column)
- Solves the stale-failure case

Cons:

- Requires extending `app-document-upload`'s internal table (or `UploadedFile` shape) to surface a failure flag — bleeds domain concerns into a generic widget. Could be gated behind an opt-in input like `[showExtractionStatus]` to keep it isolated.
- Slightly more visual noise than the current spec calls for

#### Option C — Both

Toast on transition (immediate feedback) + subtle inline indicator (persistent state). Highest UX quality, most code.

#### Option D — Defer indefinitely (not recommended)

Accept silent failures. Document the limitation in the user-facing help. Only viable if backend extraction failures are vanishingly rare in practice.

### Recommendation

Start with Option A (toast on failure transition with optional Retry action). It is:

- Smallest change (no shared-component modifications)
- Reuses existing polling
- Easy to upgrade to Option C later if users report missed failures

### Out-of-scope clarifications

- This follow-up is **only** about Overview Document Library. The wizard Create step is local file staging (no server-side processing has happened yet) and does not need any failure surface.
- The compliance items panel and viewer panel already show their own loading / error states for in-progress and failed documents — this follow-up is specifically about the Document Library list view on Overview.
- Server-side retry (`POST /compliance/projects/:projectId/documents/:documentId/process`) already exists from Phase 4. The follow-up is purely a frontend UX layer on top of it.

## Follow-up: Document Library locked-state UX

> **Status:** Deferred. To be addressed in a separate ticket.

### Problem

The Overview page's Document Library (`app-document-upload`) is currently **locked** (drag-and-drop disabled, delete button disabled, upload button disabled) while **any** document in the project is being processed. It also **stays locked** if any document ends in `processingStatus = Failed`, because the guard only distinguishes "in-flight" from "done" — not "done successfully" vs. "done with errors".

Concretely, the lock is driven by a computed flag along the lines of:

```ts
hasProcessingDocuments = computed(() =>
  this.documents().some(
    (d) => d.processingStatus === Processing || d.processingStatus === Pending,
  ),
);
```

Failed documents exit the processing states, but the overall "library is usable" signal is effectively `!hasProcessingDocuments && !hasFailedDocuments` in practice — and a failed document cannot be removed because the delete button is part of the locked UI. The user's only escape is to refresh and hope state recovers.

### Failure modes this creates

1. **Long happy-path uploads block the library.** A user uploads 5 PDFs; until the last one finishes extracting, they cannot add or remove any files.
2. **One failed doc poisons the library.** If the 5th PDF fails to extract, the library stays locked indefinitely — the user cannot delete the failed doc to unstick themselves, and cannot upload new documents.
3. **No signal to the user about why it's locked.** There is no inline explanation ("3 of 5 documents still processing…" or "1 document failed to process — delete to continue").

### Suggested approaches (pick one in the follow-up ticket)

#### Option A — Only lock per-row; keep library-level actions always enabled (recommended)

- Upload (drag-and-drop) is **always enabled**.
- Delete is **always enabled** per row, including for `Processing`, `Pending`, and `Failed` documents (delete cascades cleanly — backend and DB already handle mid-extraction cancellation via row deletion).
- Only the **per-row content** that depends on extraction (e.g. clicking to open the viewer) shows a per-row spinner/"Processing…" state.

Pros:

- User is never stuck. They can always add or remove documents.
- Failed docs can be removed without refreshing.
- Matches the Create step's UX (which has no lock at all).

Cons:

- Requires the backend to tolerate deletion during extraction (verify the compliance listener handles `ON DELETE CASCADE` mid-`AutoTagComplete` — likely already fine because the listener looks up the document by ID before updating).

#### Option B — Lock only during in-flight; auto-unlock on `Failed`

- Keep current locking behaviour during `Processing` / `Pending`.
- On `Failed` transition, **unlock** the library so the user can delete the failed doc.
- Pair with the toast from the "Document processing status visibility" follow-up so the user knows why it unlocked.

Pros:

- Preserves the current "don't touch while working" safety model.
- Unsticks the failure case without extra UI.

Cons:

- Still blocks the user during long happy-path uploads (no ability to queue more).
- Two follow-ups become coupled (locking + toast).

#### Option C — Explicit lock reason banner + partial unlock

- Keep the lock, but surface an explicit banner: "3 of 5 documents processing — library edits disabled" or "1 document failed — delete it to continue."
- Allow delete on `Failed` rows only.

Pros:

- Clear user communication.
- Safer than full unlock.

Cons:

- Most UI work of the three options.
- Still blocks uploads during happy-path processing.

### Recommendation

Lean toward **Option A**. The Create step already allows unrestricted add/remove during local file staging, so Overview matching that behaviour is consistent. The main verification needed is confirming the backend tolerates deletion mid-extraction (compliance listener + Service Bus `AutoTagComplete` for a deleted document should be a safe no-op via the existing "document not found / skip" guard pattern).

### Out-of-scope clarifications

- This follow-up is only about the Overview Document Library lock. The wizard Create step has no equivalent lock (it operates on local files pre-upload) and needs no change.
- This follow-up is **distinct from** "Document processing status visibility" above, but they naturally pair: visibility tells the user _what_ happened, unlock lets them _act_ on it. A single combined ticket is reasonable.

## Follow-up: Wizard Preview delete button is not wired

> **Status:** Deferred. Pre-existing gap — **not introduced by v3.5**. To be addressed in a separate ticket.

### Problem

`create-project-wizard.component.html` embeds `<app-project-snapshot-content>` in its Preview step to render a read-only-ish summary of the project before the user clicks Finish. That summary includes the Document Library section, which renders `<app-document-upload>` with a trash icon per file row (and has since the component was introduced).

The wizard's Preview step does **not** bind the `(deleteDocument)` output from `<app-project-snapshot-content>`. The trash icon therefore fires into the void — the file appears deleted visually for the duration of a click animation, nothing happens server-side, and the next change-detection cycle re-renders the row from `uploadedFiles()`. The user sees no feedback and no deletion occurs.

This gap existed before v3.5 — the same missing wire was present with the bespoke `compliance-document-table` — and was surfaced during the v3.5 plan-compliance review. v3.5 neither introduces nor fixes it.

### Scope

The Preview step operates on documents that are already uploaded to the backend (the user has progressed through Create → files were POSTed via `uploadSourceDocument` → they're now reviewing before clicking Finish to trigger auto-tag). Deleting from Preview therefore has the same semantics as deleting from Overview: it must call `ComplianceApiService.deleteSourceDocument()` and refresh the project.

### Suggested approaches (pick one)

#### Option A — Wire `(deleteDocument)` through the wizard (recommended)

In `create-project-wizard.component.html`, add `(deleteDocument)="onDeleteDocumentPreview($event)"` to the `<app-project-snapshot-content>` binding. Implement `onDeleteDocumentPreview(documentId: string)` on the wizard component — same pattern as `compliance-project-overview-page.onDeleteDocument`: call the delete API, refresh the project, update local staged-file state so the Preview re-renders correctly.

Pros:

- Mirrors Overview delete flow exactly (consistency)
- Smallest change
- User can correct a wrongly-uploaded file from Preview without navigating back to Create

Cons:

- Deleting from Preview then clicking Finish must correctly skip the just-deleted document during auto-tag trigger (verify `processDocuments` / `onFinish` uses the refreshed list, not a cached one)

#### Option B — Hide the delete column in Preview

Add a `[showDeleteAction]` input to `app-document-upload` (default `true`). In Preview, bind `[showDeleteAction]="false"` so the actions column is suppressed. The user must navigate back to Create (via the pencil edit button, which already works) to remove a file.

Pros:

- Zero risk to the Finish flow
- Clarifies Preview's intent as read-only-for-deletions

Cons:

- New input on a shared component for a narrow use case
- Overview would keep the column; two code paths diverge slightly
- Extra click for the user to remove a wrongly-uploaded file

#### Option C — Disable the delete button in Preview

Add a `[disableDeleteAction]` input (or reuse the existing `[disabled]` on `app-document-upload`, which already disables the whole drop zone + delete — but that also disables upload, which Preview currently allows since v3.5 added `(filesSelected)` wiring). A more targeted flag would be needed.

Pros:

- Leaves the column visible (affordance) but prevents the no-op click

Cons:

- Confusing UX (visible disabled buttons with no explanation)
- Needs a new input on the shared component

### Recommendation

Option A. The Preview step is conceptually "last chance to review before committing to auto-tag" — allowing both upload and delete here is consistent with that framing and with how v3.5 already wired `(filesSelected)` through. The key verification is that `onFinish()` reads from the post-refresh document list so a just-deleted document isn't submitted to the auto-tag queue.

### Out-of-scope clarifications

- This follow-up is only about the wizard Preview step's delete button. Overview delete is already wired correctly as of v3.5.
- This follow-up is independent of the "Document processing status visibility" and "Overview Document Library lock" follow-ups above — all three can be shipped together or separately.
