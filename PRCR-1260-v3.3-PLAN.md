## Problem Statement

The **Document Viewer Panel** in the Compliance List Creation workflow (`document-viewer-panel.component`) currently displays **mock/hardcoded text lines** instead of actual document content. The `mapProjectDocumentsToViewerDocuments()` utility in `compliance-document.utils.ts` maps project documents onto `MOCK_COMPLIANCE_DOCUMENTS` line data (with a TODO to replace it).

Users need to:

1. View the **real content** of each uploaded source document in the right-hand panel, with layout that **matches the look of the original document**.
2. **Highlight text** within the rendered document and create compliance items (tagging) from the selection — even when auto-tagging has not been run.
3. **Edit both auto-tagged and manually created compliance items** (title, text, etc.) while viewing the highlighted source passage for context.
4. Switch between multiple documents via the existing dropdown.
5. **Add new documents** to an existing project from the overview page's Document Library (post-wizard), with auto-tagging running automatically on only the new documents after conversion completes.
6. **Delete documents** from the library, with all associated compliance items automatically removed from the database.

The viewer must support **PDF, DOCX, and XLSX** files. Docling already converts all three formats to HTML uniformly, so one rendering path handles all file types.

### Document lifecycle context

- The **wizard** (Create → Preview → Finish) already triggers auto-tagging on all uploaded documents when the user clicks Finish (`onFinish()` calls `processDocuments(projectId, true)` fire-and-forget). This behaviour is correct and does not change.
- After finishing the wizard, the **overview page** includes a Document Library section. Documents uploaded here must be auto-tagged automatically after upload — there is no "Finish" button.
- Deleting a document from the overview page must cascade-delete all associated compliance items (already handled by `ON DELETE CASCADE` in the DB).

## Key Architectural Observations

### How HTML is produced today

1. **Upload**: Frontend uploads file via `POST /compliance/projects/:project_id/documents` → stored in MinIO at `compliance_documents.minio_object_key`. **Post-merge:** the upload endpoint (`uploadSourceDocument`) now also resets `autotag_processing` to `OutlineStatus.n` (`'Not Started'`) and emits `compliance.auto.tag`, so new documents are auto-tagged immediately after upload.
2. **Auto-tag trigger**: `POST /compliance/projects/:project_id/documents/process` → NestJS emits `compliance.auto.tag` event → `ComplianceListener.autoTagHandler` sends one Service Bus `AutoTagRequestMessage` per document. **Post-merge:** the listener now skips documents where `processingStatus !== PENDING_EXTRACTION || taggable_doc_id != null`, preventing double-submission. It also sets `processingStatus = EXTRACTING` before the Service Bus call.
3. **Python worker**: Downloads file from MinIO → Docling converts to HTML → `strip_html_metadata` removes `<style>`, `<script>`, `<head>` → segments HTML → classifies tags → **uploads stripped HTML** to MinIO at `{parent_path}/{output_folder}/{stem}.html` (where `output_folder` defaults to `"output"` via `_resolve_output_folder`, configurable per-message or via `MINIO_TAGGING_FOLDER_OUT`) → sends `AutoTagComplete` with `converted_html_url`.
4. **NestJS handler**: `AutoTagCompleteHandler` persists tags → emits `tagging.auto-tag-complete` event with `convertedHtmlUrl`.
5. **Compliance listener**: `handleAutoTagComplete` marks document ready/failed, loads `DocumentTag` rows, creates `ComplianceItem` rows — but **does not store `convertedHtmlUrl`**.

### Tag config system

The `tag_configs` table stores one row per product code (unique constraint on `product_code`). Each row has `tag_schema` (JSONB), `tag_ui` (JSONB, nullable), `segmentation_strategy`, `prompt_name`, and `rule_pattern`. The backend serves tag configs via `GET /tagging/tag-configs?product_code=...`.

**[v3.4 — PRCR-1421 landed]** The `tag_ui` column (added by PRCR-1421) holds a normalized UI-only display contract (`{ tags: [{ id, name, color }] }`) separate from `tag_schema` (which drives backend classification). All four products are seeded with `tag_ui` — compliance has `{ "tags": [{ "id": "compliance_item", "name": "Add Compliance Item", "color": "#CCE5FF" }] }`. The compliance `tag_schema` still has **no** `tags` or `categories` array (only pipeline fields), but this no longer matters because `doc-shell` reads `tag_ui` first.

On the frontend, `TaggingService` maintains a signal-based in-memory cache (`tagConfigsByProduct`) populated by callers via `setTagConfig(productCode, config)` / `clearTagConfig(productCode)`. Consumers read via `getTagConfigSnapshot(productCode)`. `PostLoginEffectsService` currently preloads tag configs for `TEMPLATE_GENERATOR` and `PROPOSAL_WRITER` at login — **not** `COMPLIANCE` (compliance is a separately-gated feature, so its config should be loaded lazily). **[v3.4]** `TaggingService` now resolves display metadata from `tag_ui` first, falling back to `tag_schema` only when `tag_ui` is null.

The `doc-shell` component accepts a `[tagConfig]` input (type `TagConfigResponse | null`). It does **not** inject `TaggingService` — it relies entirely on the input. **[v3.4]** Inside `buildMenuConfig()`, the shell now tries `tag_ui.tags[]` first via `resolveTagUiTags()`. Only if `tag_ui` is absent/empty does it fall back to `resolveSchemaTags(tagSchema)` which checks `tagSchema.tags` then `tagSchema.categories`. Since compliance `tag_ui` is seeded, `doc-shell` will find it and build the context menu correctly — no `tag_schema` modification is needed.

### Offset semantics

`compliance_items.document_start_line` / `document_end_line` are populated from `DocumentTag.start_offset` / `end_offset` via the compliance listener. Despite the "line" naming, these are **character offsets into the stripped HTML string**, not line numbers. The same offset space applies to the HTML served by the new content endpoint.

### Existing cascade constraints (DB)

- `compliance_items.source_document_id → compliance_documents(id) ON DELETE CASCADE`
- `compliance_project_documents(document_id) → compliance_documents(id) ON DELETE CASCADE`
- `compliance_checks.compliance_item_id → compliance_items(id) ON DELETE CASCADE`
- `compliance_item_evidence.response_document_id → compliance_documents(id) ON DELETE CASCADE`

Deleting a `compliance_documents` row automatically removes all associated items, checks, and evidence.

### Existing document deletion via PATCH

The `PATCH /projects/:id` endpoint (`updateProject`) already supports batch document deletion via a `documentsToDelete` property on the DTO. The wizard uses this when editing a project to remove documents the user un-selected. The implementation validates project-scoping, deletes `ComplianceDocument` rows (DB cascade handles items, checks, evidence), and cleans up `TaggableDocument` rows. It does **not** delete the corresponding MinIO objects (original files or converted HTML). This is intentional — files in MinIO are retained for traceability even after the database record is deleted.

The `PATCH /projects/:projectId/responses/:responseId` endpoint also has a `documentsToDelete` property, but that only unlinks response documents from a response (deletes `compliance_response_documents` join-table rows) — it does **not** delete the `ComplianceDocument` row itself.

## Assumptions

1. Docling-generated stripped HTML is **safe to render**. The shared `app-doc-shell` component already handles sanitisation via DOMPurify + `bypassSecurityTrustHtml`, matching the Proposal Writer's approach.
2. The stripped HTML uploaded by the Python auto-tag worker is the **same** HTML we serve in the viewer — no secondary conversion needed.
3. The viewer only needs to display document content **after** auto-tagging has been triggered (wizard Finish or overview upload). A "Processing…" placeholder for in-progress documents is acceptable for v1.
4. The existing `pageNumber` / `totalPages` fields on `ComplianceSourceDocument` can be dropped — Docling HTML does not inherently have page breaks.
5. The shared `app-doc-shell` component's offset algorithm (`buildFlattenedOffsetsByTextNode`) is compatible with the Python auto-tag pipeline's `extract_text_with_mapping()` — both use the same block selectors and separator. This is explicitly documented in the Python code comments.
6. `app-doc-shell` is exported by `SharedComponentsModule`, which `ComplianceModule` already imports. No additional module import is needed.
7. **Post-merge:** `doc-shell` now rebuilds flattened offsets after each highlight DOM mutation, fixing overlapping highlight rendering. This means compliance items with overlapping offsets will render correctly without custom workarounds.
8. **Post-merge:** `InlineTag.isManualPosition` has been removed from the shared type. The plan previously noted setting `x`/`y` to `0`; this is still correct, but `isManualPosition` no longer exists and should not be referenced.

Comments in the Python code explicitly reference the UI constants. This means `app-doc-shell` natively handles compliance item highlighting with no custom offset logic. |
