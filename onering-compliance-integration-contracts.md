## 1. ONERING-side contracts

### 1.1 `compliance_review:build_steps` step factory

Module path: `arc_agent_writer.factories.compliance_review`. Exposed CLI usage: `python -m arc_agent_writer.cli run --steps-factory arc_agent_writer.factories.compliance_review:build_steps`.

**Function signature** (Python):

```python
# ONERING/arc_agent_writer/factories/compliance_review.py
from typing import Any, Sequence
from arc_agent_writer.orchestrator import Step

def build_steps(config: dict[str, Any]) -> Sequence[Step]:
    """
    Thin step graph for Compliance: ingestion → chunk_plan →
    pipelines.requirements → pipelines.ui_projection_requirements.

    Bypasses metadata-derived dependencies (instructions, attachments,
    structure, evaluation) by supplying empty/default UI projections
    for those upstream tabs where pipelines.requirements consumes them.

    Returns ordered Step list compatible with Orchestrator.run().
    """
```

**Step graph** (in execution order):

1. `ingestion.docling_parse`
2. `ingestion.ocr_fallback`
3. `ingestion.canonical_markdown`
4. `ingestion.line_numbering`
5. `ingestion.chunk_plan`
6. `pipelines.requirements`
7. `pipelines.ui_projection_requirements`

**Invariants:**

- Output artifact: `AGENT_RUNS/{run_id}/pipelines/ui_projection_requirements/ui_projection_requirements.json`
- Per-call artifacts: `AGENT_RUNS/{run_id}/llm_calls/{step}/{prompt}/{call_id}/`
- Manifest: `AGENT_RUNS/{run_id}/manifest.json` includes only the seven steps above
- Resumable: each step writes its own checkpoint under `AGENT_RUNS/{run_id}/checkpoints/`

**Validation requirement:** factory must run end-to-end on the staging fixture set without metadata/structure/evaluation outputs available. Any consumer in `pipelines.requirements` that currently reads those projections must accept empty defaults; required behavioral changes are documented in the Phase 1 PR description.

### 1.3 `ui_projection_requirements.json` v1 JSON Schema

Location: `ONERING/specs/ui_projection_requirements.schema.json`. Sample fixture: `ONERING/arc_agent_writer/tests/fixtures/ui_projection_requirements.sample.json`.

**Schema (JSON Schema draft-07):**

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://onering.rohirrim.ai/specs/ui_projection_requirements.v1.json",
  "title": "UI Projection: Requirements",
  "type": "object",
  "required": ["schema_version", "run_id", "project_id", "requirements"],
  "additionalProperties": true,
  "properties": {
    "schema_version": { "type": "string", "const": "1" },
    "run_id": { "type": "string", "format": "uuid" },
    "project_id": { "type": "string", "format": "uuid" },
    "generated_at": { "type": "string", "format": "date-time" },
    "requirements": {
      "type": "array",
      "items": { "$ref": "#/definitions/Requirement" }
    }
  },
  "definitions": {
    "Requirement": {
      "type": "object",
      "required": [
        "arc_requirement_id",
        "source_document_id",
        "title",
        "text",
        "line_start",
        "line_end",
        "stable_hash"
      ],
      "additionalProperties": true,
      "properties": {
        "arc_requirement_id": {
          "type": "string",
          "description": "Engine-side stable identifier. Opaque to UI. Used as dedupe key on re-run."
        },
        "source_document_id": {
          "type": "string",
          "format": "uuid",
          "description": "compliance_documents.id"
        },
        "title": { "type": "string", "maxLength": 500 },
        "text": { "type": "string" },
        "outline_number": { "type": ["string", "null"], "maxLength": 100 },
        "section_name": { "type": ["string", "null"], "maxLength": 255 },
        "line_start": { "type": "integer", "minimum": 0 },
        "line_end": { "type": "integer", "minimum": 0 },
        "stable_hash": {
          "type": "string",
          "pattern": "^[0-9a-f]{64}$",
          "description": "sha256 hex over canonicalized requirement text. Inputs: text after whitespace-collapse, lowercase, NFC-normalized. MUST be stable across re-parses, OCR variance, and Docling version bumps. Inputs MUST NOT include line_start/line_end or document-level offsets. Used as the sole stable-key match field on re-run carry-over."
        }
      }
    }
  }
}
```

**Versioning rule.** `schema_version: "1"` allows additive changes (new optional fields, ignored by `additionalProperties: true`). Breaking changes — rename, type change, removal, semantic shift of an existing field, or making a previously-optional field required — bump `schema_version` to `"2"` and ship a new schema file. The rohan_api validator rejects unknown `schema_version` values with `OneringSchemaError`. rohan_api consumes only fields it knows about; unknown fields pass validation silently. Strictness lives at the `schema_version` field, not at field-set closure.

### 1.4 Airflow pool sizing (Helm chart)

File: `ONERING/helm/onering-airflow/values.yaml` (and per-env overrides).

| Pool | Current | Phase 1 | Notes |
|------|---------|---------|-------|
| `llm_extraction_pool` | 2 slots | **4 slots** | Shared by all DAGs; Compliance is third concurrent consumer. |
| `llm_aggregation_pool` | 1 slot | **2 slots** | Same rationale. |

`airflow-init` job creates both pools. New `compliance_pool` is **not** introduced in Phase 1; deferred until measurement spike shows starvation.

**Sizing model (Phase 1).** Capacity calc, not vibes: assume the pilot org runs 2 concurrent extractions during business hours, proposal launches ≤ 1 concurrent, discovery 1 concurrent. Each requirements extraction consumes 1 `llm_extraction_pool` slot for ~10–20 min wall clock; aggregation step consumes 1 `llm_aggregation_pool` slot for ~3–5 min. Steady-state demand ≤ 4 extraction slots and ≤ 2 aggregation slots with 50 % headroom. Re-evaluate before Phase 13 batch-enable: if Grafana panel "Slot wait time" exceeds 60 s p95 during pilot soak, bump to 6 / 3 before broader rollout. Numbers checked into `slo.md` (Phase 11) so future capacity changes are diff-able.

### 1.6 Git-sync ref pin policy

ONERING DAGs are mounted into Airflow pods via git-sync. From Phase 3 onward, `gitSync.ref` in `helm/onering-airflow/values-{dev,staging,prod}.yaml` MUST be a tag or commit SHA — never `main`. This protects dual-path prod from engine-team commits during the integration window.

| Env | Ref source | Bump trigger |
|-----|------------|--------------|
| `values-dev.yaml` | Latest engine release tag | Engine team bumps on tag cut |
| `values-staging.yaml` | Engine release tag staged for next prod | Promoted from dev after staging soak |
| `values-prod.yaml` | Engine release tag matched to a rohan_api release tag | Promoted from staging after prod readiness sign-off |

Bump procedure: PR touching only the `gitSync.ref` line. Title format `chore(onering-airflow): bump engine ref to <tag>`. SRE approval required for prod values. Phase 3 lands the initial pin; Phase 13 final step bumps prod ref to the engine release matching `onering-compliance-v1` rohan_api tag.

**ONERING submodule pin (rohan-python-api side).** Phase 1's primary consumption path is Airflow git-sync (covered above). The proposal also calls for pinning the ONERING submodule at `rohan-python-api/backend/arc_agent_writer/` to a specific commit. Phase 1 does not touch rohan-python-api code (the Service Bus path stays live during dual-path), so the submodule pin is a low-cost hygiene step: pin to the same engine release tag used in `values-prod.yaml`, bump in the same PR. Phase 3's wrapper-layer work (out of scope here) consumes this pin. Filed as Phase 13 step 13.6 (one-line bump).

### 2.3 Service method

**`OneringPipelineService` (`src/onering/services/onering-pipeline.service.ts`):**

```typescript
/**
 * Trigger an arc_compliance_review DAG run for a project's source documents.
 * Creates an or_pipeline_runs row, calls Airflow trigger, returns run handle.
 */
async triggerComplianceReview(
  orgId: string,
  userId: string,
  projectId: string,
  documentIds: string[],
  options?: { responding_company?: string; llm_mode?: string; verbose?: boolean },
): Promise<{
  arc_run_id: string;
  dag_run_id: string;
  status: RunStatus;
}>;
```

**Behavior:**

1. Resolve `organization_id` from `orgId`. **Org mapping policy.** If `OneringOrgLookupService` is a pass-through (rohan org id == ONERING tenant id), drop the indirection in Phase 6 and use `orgId` directly. If it's a real mapping table, document the bootstrap path: row created on org provisioning via `OneringOrgProvisioningService.ensureOrgMapping(orgId)`, called from the org-create handler. Open Q resolved during Phase 6.
2. Generate `arc_run_id` (UUID) up front.
3. Expand `documentIds` if empty: `documentIds.length === 0 ? loadAllProjectSourceDocs(projectId) : documentIds`. Empty MUST NOT be stored — audit fidelity.
4. Insert `or_pipeline_runs` row: `{ organization_id, project_id, document_ids, arc_run_id, airflow_dag_id: 'arc_compliance_review', run_type: COMPLIANCE_REVIEW, status: PENDING, total_steps: 7 }`. `project_id` and `document_ids` written to typed columns added in Phase 5.
5. Call `OneringAirflowClientService.triggerDagRun('arc_compliance_review', conf)` where `conf` is `ComplianceReviewConf`.
6. **Idempotent on Airflow 409.** If Airflow returns 409 (duplicate `dag_run_id`), call `getDagRun(airflow_dag_id, expected_run_id)`, treat as success, and use the returned handle. This protects against transport-error retry storms.
7. Update `or_pipeline_runs.airflow_dag_run_id` from response, set status to `QUEUED`.
8. Return `{ arc_run_id, dag_run_id, status }`.

**Errors:** `OneringAirflowError` (502 via `OneringExceptionFilter`) on trigger failure; `or_pipeline_runs` row marked `FAILED` with error captured.

**Polling cadence (refreshRunStatus).** The reconciler driving `or_pipeline_runs.status` from Airflow polls per-run at:

| Run state | Interval |
|-----------|----------|
| `PENDING`, `QUEUED` | 15 s |
| `RUNNING` | 5 s |
| `MATERIALIZING` | poll stops (materializer is in-process and synchronous; reconciler hands off) |
| `SUCCESS`, `FAILED` (terminal) | poll stops |

UI polling (`GET /onering/runs` at 10 s) layers on top — UI sees the last-polled status from the DB. Reconciler interval bounded explicitly so UI does not stall on completion for more than ~5 s.

**Stuck-run reaper.** A separate cron (every 5 min) finds runs with `status IN ('PENDING','QUEUED','RUNNING')` AND `started_at < now() - INTERVAL '90 minutes'`. For each: re-fetch Airflow status; if Airflow says terminal, transition the row; if Airflow says still running but `started_at < now() - INTERVAL '180 minutes'` (hard cap), mark `FAILED` with code `STUCK_RUN_TIMEOUT` and call `killDagRun(airflow_dag_run_id)` (this is the M6 caller — `killDagRun` is added to `OneringAirflowClientService` in Phase 6 alongside the reaper). Reaper logs an audit row to `compliance_audit_log` with `action='reaper_force_failed'`. Admin force-cancel endpoint (Phase 8) covers the < 90 min case.

**Org-membership consistency at materialize time.** The materializer (Phase 7 Phase A) MUST verify that the JSON's top-level `run_id` matches `or_pipeline_runs.arc_run_id`, the JSON's `project_id` matches `or_pipeline_runs.project_id`, and that the run row's `organization_id` matches the project's `organization_id`. Mismatch → `OneringSchemaError`. Defends against an attacker triggering the DAG manually with a victim's `expected_run_id`: the JSON they cause to be written will not match the rohan_api-stored run row.

### 4.2 Materializer behavior

Method: `ComplianceOneringMaterializerService.materialize(arcRunId, dagRunId)`. Called by the polling reconciler on terminal `SUCCESS`.

Split strictly into two phases. **Phase A** (read & validate) does all network IO outside any DB transaction. **Phase B** (mutate) is a single short transaction containing only DB ops.

**Phase A — Read & validate (no DB transaction).**

A1. `SELECT` `or_pipeline_runs` row by `arc_run_id` (no lock). Reject if missing — defends against materializer being invoked for a `arc_run_id` that rohan_api never inserted (out-of-band DAG trigger).
A2. Resolve `project_id`, `document_ids`, `organization_id` from typed columns on the run row.
A3. Load `ui_projection_requirements.json` from MinIO via `OneringArtifactService`.
A4. Schema-validate via `UiProjectionRequirementsValidatorService`. On failure: mark run `FAILED` in a separate single-row UPDATE, no items touched, exit.
A4a. **Cross-check JSON against run row.** Verify `json.run_id == or_pipeline_runs.arc_run_id` AND `json.project_id == or_pipeline_runs.project_id`. SELECT `compliance_projects.organization_id` and verify it matches `or_pipeline_runs.organization_id`. Mismatch → `OneringSchemaError`. Closes the out-of-band-trigger attack: even if an attacker triggers the DAG with a victim's `expected_run_id`, the JSON they cause to be written cannot match all three fields at once.
A5. `SELECT` current `compliance_documents` for the project (single query). Compute drop-set: requirements whose `source_document_id` no longer exists at validate time.

**Phase B — Mutate (single DB transaction, no network IO).**

B1. `BEGIN`.
B2. Re-check idempotency: `SELECT materialized_at FROM or_pipeline_runs WHERE arc_run_id = $1 FOR UPDATE`. If non-null, `ROLLBACK` and return (no-op).
B3. Match (incoming minus drop-set) against existing `compliance_items` for the project by `(source_document_id, stable_hash)`. `arc_requirement_id` is NOT a match key. `line_start` / `line_end` are NOT match keys (shift on re-parse).
B4. UPDATE matched rows: `arc_run_id`, `arc_requirement_id`, `complianceItemText`, `complianceItemTitle`, `outlineNumber`, `sectionName`, `documentStartLine`, `documentEndLine`. Preserve `status`, `reviewed_by`, `userNotes`, `lineItemNumber`.
B5. INSERT unmatched incoming rows. Acquire PRCR-1570 sequence lock for the project, allocate `lineItemNumber`, release. `extractionMethod = AUTO_EXTRACTED`, `status = PENDING_REVIEW`. Set `compliance_items.compliance_document_id = source_document_id` from the requirement (resolves the per-document UI grouping). Same column set on B4 UPDATE for matched rows that have moved between documents (rare but possible after re-parse).
B6. UPDATE existing rows where `project_id` matches AND `source_document_id IN (dag_run.conf.document_ids)` AND `arc_run_id != current_arc_run_id` AND not in match-set: `status = 'superseded'`. Scope strictly to `dag_run.conf.document_ids` regardless of length — a 3-document retry supersedes only those 3 documents' unmatched items.
B7. INSERT audit log rows for each drop computed in A5 (table: `compliance_audit_log`, action: `materializer_dropped_orphan`, payload: `{ arc_requirement_id, source_document_id }`). If `compliance_audit_log` does not exist, Phase 7 adds it; otherwise reuse.
B8. UPDATE `or_pipeline_runs.materialized_at = now()`, `status = 'SUCCESS'`. Status flip from `MATERIALIZING` → `SUCCESS` happens in the same transaction as the item writes — closes the race where UI sees terminal SUCCESS before items are queryable.
B9. `COMMIT`.

**Status transition contract.** Reconciler observes Airflow terminal SUCCESS → UPDATE row to `MATERIALIZING` (NOT `SUCCESS`) → invoke materializer → materializer's Phase B9 commits `SUCCESS` atomically with item writes. UI treats `MATERIALIZING` as a sub-state of `extracting` (banner stays up). `MATERIALIZING > 5 min` triggers a separate alert (P2) — materializer should be sub-second to ~10 s on typical 50–500-item projects; minutes means lock contention or DB regression.

**Invariant.** No MinIO reads, no Airflow calls, no `compliance_documents` lookups inside Phase B. Lock hold time bounded to compliance_items row writes only.

**Supersede strategy.** Phase 1 marks `status = 'superseded'` (extends `ComplianceItemStatus` enum). Row preserved for audit. No separate archive table. Reviewer state preservation re-evaluated post-pilot.

**Conscious deviations from `ONERING_INTEGRATION_PROPOSAL.md`.**

| Deviation | Proposal text | This plan | Reason |
|-----------|---------------|-----------|--------|
| Stable-key match composition | "stable-key match (`source_document_id` + `line_start` + canonical statement hash)" (proposal §"Multi-document mapping") | `(source_document_id, stable_hash)` only — drops `line_start` | `line_start` shifts across Docling re-parse / OCR variance, breaking carry-over. `stable_hash` made required + canonicalization contract added in §1.3. |
| Reviewer-state archival | "If no stable match, prior reviewer state is archived for audit but not propagated" | Row preserved with `status = 'superseded'`. No separate `compliance_item_archive` table. | Single-table preservation is simpler and meets the audit requirement (`status` + original row remain queryable). Re-evaluate post-pilot if reviewers want explicit archive view. |
| Document-deletion kill | "if a user deletes all source documents while a DAG is running, rohan_api fires `killDagRun(dag_run_id)`" | Implemented (Phase 7 §7.8 — see plan). Closes the proposal's gap. | Originally dropped during plan rework; restored on consistency sweep. |

**Error semantics.** Any exception in Phase B rolls back the transaction; a separate UPDATE marks `or_pipeline_runs.status = FAILED` with error code; reconciliation service (PRCR-1562) handles retry. Phase A failures (MinIO read, schema validation, cross-check) do not enter Phase B at all and mark `FAILED` directly.

**Document lifecycle invariants** (consumer side).

| Action | Effect on existing items | User remediation |
|--------|--------------------------|------------------|
| Document hard-deleted between trigger and materialize | Drop from incoming-set (B7 audit row); existing items keyed to that `source_document_id` left untouched. | None automatic. UI surfaces dropped doc names from audit log. |
| Document hard-deleted before any extraction | No items existed; no-op. | None. |
| Document REPLACED (new ID, old row deleted) | New ID has no items until next extraction. Old ID's items preserved (legacy artifacts) but flagged stale because `arc_run_id` points to a run whose `document_ids` no longer matches a live `compliance_documents` row. | UI nudges: "Document changed. Re-run extraction." Out of scope to auto-supersede — preserves reviewer state for legacy items if user wants to compare. |
| Document UPDATED in place (same ID, different content) | On next extraction, `stable_hash` changes for changed requirements → unmatched → superseded + new inserts. Unchanged requirements carry over. | Trigger re-extraction. Materializer handles the rest. |

UI banner copy for the replaced-doc case is specified in Phase 10. Backend exposes the staleness signal via `GET /onering/runs` listing — caller compares run's `document_ids` against `compliance_documents` for the project; missing docs → stale.

### 5.4 `POST /compliance/projects/:id/onering/runs/:arcRunId/cancel`

**Auth:** `AuthGuard('jwt')` + `PermissionsGuard('compliance:admin')` (org-admin only) + same two-stage org check as 5.1/5.2.

**Path params:** `id` — `compliance_projects.id`. `arcRunId` — `or_pipeline_runs.arc_run_id`.

**Response 200:**

```typescript
export class ComplianceRunCancelResponse {
  @ApiProperty() arc_run_id: string;
  @ApiProperty() dag_run_id: string;
  @ApiProperty({ enum: RunStatus }) status: RunStatus; // FAILED
  @ApiProperty() reason: string;                       // e.g. "ADMIN_FORCE_CANCEL"
}
```

**Behavior:**

1. Load `or_pipeline_runs` by `(organization_id, project_id, arc_run_id)`. 404 on miss.
2. If `status` already terminal: 409 `RUN_ALREADY_TERMINAL`.
3. Call `OneringAirflowClientService.killDagRun(airflow_dag_run_id)`. Tolerate 404 from Airflow (run already gone) but propagate other errors as 502.
4. UPDATE row: `status = FAILED`, `error_code = 'ADMIN_FORCE_CANCEL'`, `ended_at = now()`.
5. INSERT `compliance_audit_log` row with `action = 'admin_force_cancel'`, `payload = { actor_user_id, arc_run_id }`.
6. Return the cancel response.

**Errors:**

| Status | Code | Condition |
|--------|------|-----------|
| 403 | `INSUFFICIENT_ROLE` | Caller lacks `compliance:admin` |
| 404 | `RUN_NOT_FOUND` | `arc_run_id` missing or not in caller's org/project |
| 409 | `RUN_ALREADY_TERMINAL` | Run already in SUCCESS or FAILED |
| 502 | `ONERING_AIRFLOW_ERROR` | Airflow killDagRun returned non-404 error |

## 6. Feature flags

### 6.1 `ONERING_COMPLIANCE`

New entry in `src/utils/feature-flags/types/featureFlags.ts`:

```typescript
export enum FeatureFlagKey {
  // ...existing...
  ONERING = 'ONERING',
  ONERING_COMPLIANCE = 'ONERING_COMPLIANCE', // NEW
}
```

**Semantics:** org-level entitlement. When enabled for an org, Compliance auto-extraction routes through `arc_compliance_review` DAG. When disabled, falls through to existing Service-Bus path. Independent from `ONERING` (which gates the broader `/onering/*` UI surface).

**Rollout:** off for all orgs at merge time. Pilot org enabled in staging, then production. Flip via `pnpm enable-flag ONERING_COMPLIANCE` per `rohan_api/CLAUDE.md`.

**Listener fence.** While `ONERING_COMPLIANCE` is on for an org, the legacy `tagging.auto-tag-complete` handler MUST short-circuit for that org's events. Without the fence, in-flight `AutoTagRequest` jobs queued before the flag flip race the ONERING materializer and double-write items. Implementation in Phase 7.3a: handler reads `organization_id` from event payload, looks up flag state via `FeatureService.isEnabledForOrg(organizationId, 'ONERING_COMPLIANCE')`, logs `compliance.listener.skipped_due_to_onering` (with `org_id`, `project_id`, `job_id`) and returns when on. Listener stays registered until Phase 13 deletes it (after the pre-delete drain check confirms no in-flight legacy jobs).
